## Ce qui a bien fonctionné

1. **3/3 stories livrées à 100%** — Architecture propre, zéro dépendance externe, base solide pour la suite
2. **Code reviews très productives** — 14 issues détectées en 3 reviews, toutes les HIGH et MEDIUM corrigées immédiatement. La review de Story 1-2 a attrapé un vrai bug de logique dans la suppression (conditions mutuellement exclusives mal gérées)
3. **Fondations réutilisables** — Story 1-1 a posé une infrastructure solide que les stories 1-2 et 1-3 ont pu exploiter directement (Story 1-3 était essentiellement un audit)
4. **Décision de structure manuelle V13 validée** — Contrôle total sur chaque fichier, pas de dépendance à un starter template obsolète
5. **Documentation des patterns V13** — Les Dev Notes de chaque story documentent les patterns et anti-patterns, créant une base de connaissances interne

## Challenges rencontrés

1. **Pièges API V13 récurrents** — 2/3 stories ont eu des problèmes de namespace (`DocumentSheetConfig`, `loadTemplates`) ou d'API dépréciée (`Math.clamped` → `Math.clamp`). La documentation V13 n'est pas toujours claire sur les changements.
2. **Incohérence model/template/i18n** — Le renommage de `damage` en `rateOfFire` dans Story 1-2 sans mise à jour du template weapon-sheet a causé une régression HIGH. Pattern dangereux : modifier un data model sans vérifier tous les consommateurs.
3. **Aucune validation runtime Foundry** — Les tasks de vérification manuelle (Tasks 8, 7, 4) restent non complétées dans les 3 stories. On a construit 3 stories d'infrastructure sans jamais valider dans un vrai FoundryVTT.
4. **Bugs critiques détectés uniquement en code review** — Sans reviews systématiques, ces bugs seraient passés en production.

## Enseignements clés

1. **Toujours vérifier la cohérence model/template/i18n** — À chaque modification d'un data model, vérifier systématiquement les templates HBS et les clés i18n associées
2. **Les code reviews sont indispensables** — Surtout avec l'API V13 qui comporte de nombreux pièges
3. **Les fondations bien pensées facilitent les stories suivantes** — L'investissement dans Story 1-1 a payé pour 1-2 et 1-3
4. **La validation runtime doit être priorisée** — Construire plusieurs stories sans tester dans l'environnement cible est risqué
